# APM Probes and Enhanced Trace Context in Cloud-Native Environment

This document discusses what **enhanced context** **APM probes** (e.g., OpenTelemetry Agent) provide for **distributed
tracing** in **cloud-native** environments, and how that context helps with **problem diagnosis**, **capacity** , and *
*stability**.

--- 

## 1. What Is an APM Probe (in the Tracing Context)

Here "probe" does **not** mean Kubernetes liveness/readiness probes. It means the **auto-instrumentation layer** in *
*APM (Application Performance Monitoring)**:

- **Form**: Attached to the application process as a Java Agent, sidecar, or SDK.
- **Role**: Automatically creates and enriches **traces/spans** with **infrastructure, runtime, and business attributes
  ** with **no or minimal code changes**.
- **In this project**: We use the **OpenTelemetry Java Agent** as the probe, **Micrometer Tracing + OTLP** to export
  traces to Jaeger, and a **Filter** to write request headers (`X-User-Id`, `X-Request-Id`, `X-Tenant-Id`) into the
  current span, giving both automatic and business context.

--- 

## 2. What Enhanced Context Probes Add to Traces in Cloud-Native

In cloud-native setups (containers, K8s, service mesh), a single request touches many components (gateway, services,
message queues, databases).

"Call graph" alone is often not enough; context such as **where it ran, which version, who called, and whether
dependencies are health** is critical. APM probes add context along these dimensions.

### 2.1 Service and Instance

| Context                | Meaning                   | Typical use                                                        |
|------------------------|---------------------------|--------------------------------------------------------------------|
| service.name           | Current service id        | Filter by service in Jaeger, latency/error stats                   |
| Instance ID / hostname | Single instance or Pod id | Distinguish replicas, pinpoint “which machine/Pod”                 |
| Version / image tag    | Deployment version        | Compare latency/errors after canary/blue-green, rollback decisions |
| Environment (env)      | dev / staging / prod      | Avoid mixing environments, capacity and SLO per env                |

Usually set via **resource attributes** (OTel Agent can set these from env, K8s downward API, etc.).

## 2.2 Infrastructure and Orchestration (Cloud-Native Specific)

| Context              | Meaning              | Typical use                              |
|----------------------|----------------------|------------------------------------------|
| Container ID         | Runtime container id | Tie to container runtime and logs        |
| Pod name / Namespace | K8s metadata         | Correlate with K8s events, logs, metrics |
| Node name            | Host node            | Node-level issues (e.g. one node slow)   |
| AZ / Region          | Cloud region         | Cross-AZ/region latency and blast radius |

In K8s this is often injected via **environment variables** (e.g., `OTEL_RESOURCE_ATTRIBUTES` or the agent's K8s
resource detector).

### 2.3 Request and Dependencies

| Context                      | Meaning                 | Typical use                                    |
|------------------------------|-------------------------|------------------------------------------------|
| HTTP method, path, status    | API semantics           | Per-endpoint P99, error rate, SLO              |
| Downstream service, address  | Called service/DB/queue | Topology, slow dependency, failure propagation |
| DB instance, statement type  | DB call metadata        | Slow-query instance/type, SQL sampling         |
| Queue, topic, consumer group | Messaging context       | Message latency and backlog, consumer issues   |

Generated by **OTel Agent instrumentation** (Servlet, JDBC, RabbitMQ, etc.) without application code.

### 2.4 Exceptions and Errors

| Context                                  | Meaning         | Typical use                               |
|------------------------------------------|-----------------|-------------------------------------------|
| Exception type, message, stack (sampled) | Error details   | See failure in trace, link to logs/alerts |
| Span status (error)                      | Marked as error | Quickly filter failed requests in UI      |

The probe or SDK sets span status and exception events when errors are caught.

### 2.5 Business (Application-Level)

| Context    | Meaning           | Typical use                                   |
|------------|-------------------|-----------------------------------------------|
| user.id    | Requesting user   | Full trace per user, reproduce “user X error” |
| tenant.id  | Multi-tenant id   | Per-tenant performance and isolation          |
| request.id | Single-request id | Correlate across services and logs            |

In this project, **TracingContextFilter** in the **customer** service reads `X-User-Id`, `X-Request-Id`, `X-Tenant-Id`
from headers and sets them on the current span; other services can reuse or centralized this.

## 3. How This Context Helps (Problem Diagnosis and Beyond)

### 3.1 Problem Diagnosis

- **Sympotom**: P99 spike or error rate increase.
- **Benefits**:
    - Use **service + instance/Pod** in traces to see which instances are slow or falling; use **Node/AZ** to see if
      it's node- or AZ-wide.
    - Use **downstream** spans to see if the bottleneck is gateway, another service, or DB; use **DB instance/statement
      type** or **messaging** context to narrow down.
    - **Exception type and stack** on the span avoid hunting by traceId in logs; once trace gives "call path + failure
      point".
    - With **user.id / request.id**, pull the full trace for "that user / request" to reproduce and verify fixes.

Richer context shortens the path from "see symptom" to "instance, dependency, error, user", and reduces MTTR.

### 3.2 Capacity and Performance

- **Context**: Service, instance, version, path, dependencies, Node/AZ
- **Benefits**:
    - Latency percentiles and QPS per service/endpoint/dependency; find hotspots and bottlenecks.
    - Per-instance/Pod load balance; per Node/AZ to see local hotpots or cross-AZ latency.
    - Version + time supports pre/post-release analytics and rollback decisions.

### 3.3. Stability and SLO

- **Context**: Error status, exceptions, service/instance/version, dependency health.
- **Benefits**:
    - Define error rate and latency SLO per service/endpoints; use **status** and **exception** in traces for accurate
      failure counts.
    - Topology and per-dependency latency/errors reveal failure propagation and support circuit-breaking and throttling.
    - Pod/Node/AZ dimensions align with K8s events and cluster monitoring for unified alerting and automation.

### 3.4 Business and Ops

- **Business context** (user.id, tenant.id, request.id) lets traces answer "who and which business request", not only "
  which service/instance and what error".
    - "Trace by user/tenant/request" for support and product to reproduce issues.
    - Correlate with logs and business events via **request.id / trace_id** for a single "trace + logs + events" view
      and better collaboration in cloud-native ops.

## 4. How This Project Implement It

- **AMP probe**: Each service's Dockerfile includes the **OpenTelemetry Java Agent**; `-javaagent` does
  auto-instrumentation and exports to Jaeger via OTLP. In K8s, set `OTEL_SERVICE_NAME`, `OTEL_EXPORTER_ENDPOINT`, and
  `OTEL_RESOURCE_ATTRIBUTES` (e.g. Pod name, Namespace) for richer resource context.
- **Business context**: **TracingContextFilters** in **customer** writes `X-User-Id`, `X-Request-Id`, `X-Tenant-Id` to
  the current span; other services can reuse or extend (e.g., from JWT or gateway headers).
- **Stack**: Traces are still exported via **Micrometer Tracing + OTLP** to Jaeger; the Agent adds automatic and
  resource/dependency/exception context, the Filter adds business context, together giving the enhanced context and
  benefits above.

## 5. Summary

| Dimension            | Context from probe                          | Benefit for diagnosis/system                             |
|----------------------|---------------------------------------------|----------------------------------------------------------|
| Service & instance   | service name, instance ID, version, env     | Pinpoint instance/version; release and capacity analysis |
| Cloud-native infra   | container, Pod, Node, Namespace, AZ         | Align with K8s/cloud; node/AZ-level diagnosis            |
| Request & dependency | HTTP semantics, downstream service/DB/queue | Topology, slow dependency, failure propagation           |
| Exceptions           | type, message, stack, span status           | Error location in one trace; lower MTTR                  |
| Business             | user.id, tenant.id, request.id (app-set)    | Trace by user/tenant/request; support and product repro  |

In a cloud-native setup, **APM probe + business attribute** gives traces full context from infrastructure to business
request, so a trace answers "when, who, which version, which dependency, what error", and delivers real gains in
diagnosis, capacity, and stability.

## 6. Probes and "Instrumentation" (Including Alibaba Terminology)

In Alibaba-sytle terminology, "non-invasive instrumentation" / "staking" means inserting **collection logic** into key
code paths **without changing business code** - like "driving stakes" at important points to capture data.

The **APM probe** is the component that does this: a Java Agent (or similar) that uses bytecode instrumentation to hook
HTTP, JDBC, RPC, messaging, etc., and produce traces and metrics.

So:

- Instrumentation / stacking = the **activity** (inserting collection points).
- Probe / Agent = the **component** that does it (e.g., ARMS probe, SkyWalking Agent, OpenTelemetry Java Agent).

Phrases like "non-invasive" or "auto instrumentation" in ARMS/SkyWalking docs refer to the same idea as "APM probe
enriching trace context" in this documentation.

## 7. ARMS, SkyWalking, and a Cloud-Native / Europe-Friendly Option

When choosing a tracing and APM-probe stack, common options are **Alibaba Cloud ARMS**, **Apache SkyWalking**, and an **
OpenTelemetry-based, cloud-native and Europe-friendly** approach. Below is a short trade-off comparison and a
recommendation for cloud-native and European compliance.

### 7.1 Alibaba Cloud ARMS

| Aspect | Notes                                                                                                                                                                                                                                     |
|--------|-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Role   | Alibaba Cloud–hosted APM; tight integration with EDAS, ACK, Log Service, etc.                                                                                                                                                             |
| Probe  | Java/Node agents; non-invasive instrumentation; same kind of context as described above.                                                                                                                                                  |
| Pros   | Turnkey on Alibaba Cloud; good integration and Chinese docs/support.                                                                                                                                                                      |
| Cons   | Tied to Alibaba Cloud; data typically in Alibaba regions (including China), which raises **EU deployment and data-localization** and trust concerns; multi-cloud or self-hosted K8s need extra thought for onboarding and data residency. |
| Fit    | Teams primarily on Alibaba Cloud and comfortable with data in Alibaba China/international regions.                                                                                                                                        |

### 7.2 Apache SkyWalking

| Aspect | Notes                                                                                                   |
|--------|---------------------------------------------------------------------------------------------------------|
| Role   | Open-source APM and tracing; Apache top-level project; can be self-hosted or via commercial distros.    |
| Probe  | SkyWalking Java Agent, etc.; auto-instrumentation; OTLP and other protocols supported.                  |
| Pros   | Open source; self-hosted; full data control; integrations and adoption in Alibaba ecosystem and others. |
| Cons   | Legacy protocol and data model; unifying with OTLP/OTel may need adapters; you own ops and upgrades.    |
| Fit    | Teams that want self-hosted APM, are familiar with SkyWalking, and can run it themselves.               |

### 7.3 Cloud-Native and Europe-Friendly: OpenTelemetry + Jaeger / Tempo (Recommended)

For a **cloud-native, vendor-neutral** stack that keeps data **in your control and in Europe** (for any region), we
recommend **OpenTelemetry (OTel)** as the single probe and protocol, with **Jaeger** or **Grafana Tempo** (or both) as
the backend:

| Aspect             | Notes                                                                                                                                                                        |
|--------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Role               | OTel is a CNCF standard for collection and protocl (OTLP); backend is your choice of open-source, self-hosted Jaeger / Tempo-no vendor lock-in.                              |
| Probe              | **OpenTelemetry Java Agent** does non-invasive instrumentation with the same kind of context as ARMS/SkyWalking (service, dependencies, exceptions, K8s), and is native OLTP |                                                                                                                                                |
| Cloud-native       | Fits K8s, Helm, operators; works with Prometheus, Loki, Grafana for a single observability stack; OTel SDKs/agents for many languages and runtimes.                          |
| Europe-friendly    | Data can stay 100% in your K8s or EU DC/region, meeting **data localization and GDPR**; no default shipment to china or US vendors.                                          |
| vs ARMS/SkyWalking | No dependency on Alibaba Cloud or SkyWalking distros; you can swap backends (Jaeger today, Tempo tomorrow) while keeping one probe and protocol (OTel).                      |

**This project** uses that path: **OTel Java Agent + OTLP export + Jaeger**. You can add **Grafana Tempo** or the full
Grafana stack later for tighter log/metric correlation without changing the probe or collection side.

### 7.4 Comparison Summary

| Option                           | Cloud-native friendly      | Europe / data localization          | Vendor-neutral    | Non-invasive probe   |
|----------------------------------|----------------------------|-------------------------------------|-------------------|----------------------|
| Alibaba ARMS                     | Good on Alibaba Cloud      | Needs region/compliance review      | No (Alibaba)      | Yes                  |
| Apache SkyWalking                | Good when self-hosted      | Yes if self-hosted                  | Yes (open source) | Yes                  |
| **OpenTelemetry + Jaeger/Tempo** | **Yes (CNCF, K8s-native)** | **Yes (fully self-hostable in EU)** | **Yes**           | **Yes (OTel Agent)** |
